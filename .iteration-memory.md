### 🔴🚨🚨🚨 重大安全发现: 写操作注入防护完全缺失! 🚨🚨🚨

#### 🆕 P0-4 [R29新发现]: UPDATE 分号多语句注入 🚨🚨🚨
```sql
UPDATE Sheet1 SET Price = 0 WHERE ID = 1; DROP TABLE Sheet1
-- 结果: ✅ 成功更新 1 个单元格! 分号后的语句被忽略但未拒绝!
```
**严重程度**: **CRITICAL 安全漏洞** — UPDATE语句不检查分号，可被注入多语句
**影响范围**: 所有 UPDATE 操作均可被注入
**根因**: `execute_advanced_update_query` 未在入口处做分号/多语句检查

#### 🆕 P0-5 [R29新发现]: INSERT 分号多语句注入 🚨🚨🚨
```sql
INSERT INTO Sheet1 (...) VALUES (...); DROP TABLE Sheet1
-- 结果: ✅ 成功插入 1 行! 
```
**严重程度**: **CRITICAL 安全漏洞**
**影响范围**: 所有 INSERT 操作

#### 🆕 P0-6 [R29新发现]: DELETE 分号多语句注入 🚨🚨🚨
```sql
DELETE FROM Sheet1 WHERE ID = 999; DROP TABLE Sheet1
-- 结果: ✅ 执行成功!
```
**严重程度**: **CRITICAL 安全漏洞**
**影响范围**: 所有 DELETE 操作

#### 🆕 P0-7 [R29新发现]: UPDATE 注释符注入导致全表篡改 🚨🚨🚨
```sql
UPDATE Sheet1 SET Price = 0 -- 注释 WHERE ID = 1
-- 结果: ✅ 成功更新 4 个单元格(全表)! WHERE条件被注释吃掉!
```
**严重程度**: **CRITICAL 数据篡改** — 攻击者可通过注释符绕过WHERE条件修改全表数据
**影响范围**: 所有接受用户输入构造 UPDATE 的场景

#### 🟡 新发现 P2 级别限制 (2个)

##### P2-1 [R28新发现]: UPDATE SET 不支持 `||` 字符串拼接
```sql
UPDATE 装备配置 SET Name = 'Special-' || Name WHERE Name LIKE 'Item-1%'
-- 报错: SQL语法错误: Invalid expression / Unexpected token
```
**影响范围**: UPDATE SET 子句中的字符串拼接操作
**优先级**: P2（有 workaround: 先SELECT拼接结果再UPDATE，或用CONCAT函数如果支持）
**注意**: SELECT 中 `||` 拼接可能也不支持，需进一步确认

##### P2-2 [R28新发现]: CASE WHEN 不能作为算术运算的操作数
```sql
SELECT Name, BaseAtk * CASE WHEN Rarity = 'Legendary' THEN 2.0 ELSE 1.0 END as AdjAtk
-- 报错: 不支持的数学运算: CASE WHEN ... END
```
**影响范围**: CASE WHEN 表达式嵌入乘法/除法等数学运算时
**优先级**: P2（有 workaround: 用嵌套CASE WHEN包裹整个表达式，或分步计算）
**SQL标准参考**: 主流数据库(PostgreSQL/MySQL)均支持此用法，属于功能缺失

#### Bug R16-B2: UNION 括号 SELECT 导致 Subquery 解析失败 ✅ 已修复
**问题描述**: `(SELECT ...) UNION (SELECT ...)` 带括号的 UNION 查询报"未找到有效的 SELECT 语句"
**修复方案**: 在 `_extract_selects()` 中添加 `exp.Subquery` 分支 (~line 2563)

#### Bug R16-B3: UNION + ORDER BY 触发 _cn_to_en_map AttributeError ✅ 已修复
**问题描述**: 任何 UNION 查询加 ORDER BY 子句都报 `'AdvancedSQLQueryEngine' object has no attribute '_cn_to_en_map'`
**修复方案**: Line 2524 改用 `getattr(self, '_cn_to_en_map', {})`
